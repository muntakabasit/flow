🎯 KEY FEATURES
═══════════════════════════════════════════════════════════════════════════

TURN BOUNDARY DETECTION
  • 250ms hangover window (allow natural pausing)
  • Dynamic thresholds based on pause tolerance
  • Normal: 700ms (fast conversations)
  • Long: 1100ms (natural speakers)

BARGE-IN SUPPORT
  • Web: VAD-based (speech power detection)
  • iOS: Level-based (audio peak detection)
  • Threshold: 0.015 (web) / 0.04 (iOS)
  • Debounce: 300ms (prevent multiple triggers)
  • Effect: Immediate TTS interruption + mic resume

MESSAGE PROTOCOL
  • speech_start: User begins speaking
  • speech_end: User finishes (after hangover)
  • barge_in: User interrupts TTS playback
  • Both directions: Client → Server & Server → Client

SETTINGS

  • Persists to localStorage (web) / UserDefaults (iOS)
  • Real-time effect on turn boundaries
  • Help text explaining use cases

🎓 KEY CONCEPTS
═══════════════════════════════════════════════════════════════════════════

HANGOVER WINDOW (250ms)
  • After speech ends, wait 250ms before finalizing
  • Allows user to continue after natural pauses
  • Then evaluate against configured threshold

PAUSE TOLERANCE
  • Normal: Turn finalizes after 700ms silence
  • Long: Turn finalizes after 1100ms silence
  • Applies AFTER hangover window expires
  • User can switch anytime (even during session)

BARGE-IN
  • User speaks while TTS is playing
  • Immediate detection (threshold-based)
  • TTS stops, mic resumes
  • Debounce prevents rapid re-triggers (300ms)

MESSAGE FLOW
  • Client VAD detects speech → speech_start message
  • Client VAD detects silence (after hangover) → speech_end message
  • User interrupts TTS → barge_in message
  • Server logs and handles all three types
